Textzy SMS API Reference

Production-grade reference for Textzy SMS integrations covering simple public URL mode, authenticated tenant SMS send, DLT compliance, Tata gateway mapping, sender and template registries, webhook callbacks, and rollout controls.

Integration modes
Simple URL API Authenticated Tenant API DLT-ready Tata-compatible
Base URL
https://textzy-backend-production.up.railway.app

1. Public SMS API No bearer token

Use this for ERP, CRM, admin-panel, or lightweight partner integrations where you want a browser-friendly or server-friendly request model without session login, while still keeping each tenant on its own credential set.

GET Request

GET https://textzy-backend-production.up.railway.app/api/public/messages/send
  ?recipient=919999999999
  &msg=Your%20approved%20DLT%20message%20text
  &user=MONEYART
  &pswd=YOUR_PASSWORD
  &apikey=YOUR_API_KEY
  &channel=sms
  &sender=MNYART
  &PE_ID=1601100000000006533
  &Template_ID=1207171593687982329
  &tenantSlug=moneyart

POST Request

{
  "recipient": "919999999999",
  "message": "Your approved DLT message text",
  "user": "MONEYART",
  "password": "YOUR_PASSWORD",
  "apiKey": "YOUR_API_KEY",
  "tenantSlug": "moneyart",
  "channel": "sms",
  "sender": "MNYART",
  "peId": "1601100000000006533",
  "templateId": "1207171593687982329",
  "idempotencyKey": "sms-20260308-0001"
}
Use the backend URL only. For India DLT, sender, PE_ID, Template_ID, and message text must match approved policy.

2. Required and Optional Fields

FieldTypeUse
recipientstringMobile number with country code
message / msgstringSMS text
tenantSlugstringRequired target tenant slug
userstringTenant-scoped API username
password / pswdstringAPI password
apiKey / apikeystringAPI key
senderstringDLT-approved sender ID
peId / PE_IDstringDLT entity ID
templateId / Template_IDstringDLT template ID
idempotencyKeystringOptional request de-duplication key
Platform owner provisions public API credentials per tenant. Requests are validated against the selected tenantSlug only, and optional IP whitelist rules can be enforced per tenant.

3. Authenticated Tenant SMS Send

Use the authenticated tenant API when SMS sends originate from Textzy web, mobile, or desktop flows and need billing, audit, and inbox workflow integration.

{
  "recipient": "919999999999",
  "channel": "Sms",
  "body": "Your OTP is 123456",
  "smsSenderId": "MNYART",
  "smsPeId": "1601100000000006533",
  "smsTemplateId": "1207171593687982329"
}
Requires authenticated tenant session, selected project, cookie-based first-party auth, and an idempotency key on send.

4. Tata Gateway Mapping

https://smsgw.tatatel.co.in:9095/campaignService/campaigns/qs
  ?recipient=<MobileNo>
  &dr=false
  &msg=<Message>
  &user=<Username>
  &pswd=<Password>
  &sender=<SenderAddress>
  &PE_ID=<PEID>
  &Template_ID=<TemplateID>

Textzy keeps the Tata transport centralized while your integration payload supplies the recipient, sender, PE_ID, template ID, and final approved message text.

6. DLT and Segment Rules

Sender DLT-approved sender ID required.
PE_ID DLT-approved entity ID required.
Template_ID DLT-approved template ID required.

Segment Logic

  • English GSM: 1-160 chars = 1 SMS, multipart uses 153 chars/segment
  • Unicode / Regional: 1-70 chars = 1 SMS, multipart uses 67 chars/segment

7. Webhooks

DLR Webhook

POST /api/sms/webhook/tata?tenantSlug=<tenantSlug>

Inbound STOP / Unsubscribe

POST /api/sms/webhook/tata/inbound?tenantSlug=<tenantSlug>

Supported opt-out keywords: stop, unsubscribe, optout, cancel, quit, end.

8. Response Patterns

Success

{
  "jobId": "5d64f8bf-4c1f-4e59-92a9-4a0f5a8c992e",
  "message": "Accepted"
}

Error

{
  "message": "Invalid authorization.",
  "code": "401"
}
  • 400 request rejected
  • 401 invalid authorization
  • 403 access denied
  • 422 DLT validation issue
  • 429 rate limit exceeded
  • 503 gateway temporarily unavailable

9. Testing and Rollout

Use POST /api/platform/settings/sms/test before go-live to validate Tata configuration and live parameter mapping.

Recommended production flow: secure credentials, use HTTPS only, validate DLT identifiers, watch DLR callbacks, and test every sender/template pair before routing customer traffic.
